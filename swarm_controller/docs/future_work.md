# Future work — улучшения после базовой реализации MPC

Этот документ собирает архитектурные, алгоритмические и реализационные
улучшения, которые обсуждались, но **сознательно отложены** до момента, когда
базовый ACC MPC заработает. Каждый пункт содержит:
- **Контекст** — как сейчас, что не идеально
- **Предложение** — куда двигаться
- **Приоритет** — низкий / средний / высокий
- **Зависит от** — что должно быть готово до этого

---

## Архитектурные

### 1. Body-frame координаты в Telemetry

**Контекст.** Сейчас `Telemetry.msg` несёт `x, y, theta, peer_x, peer_y` в
глобальной системе координат (`map` frame). Это требует:
- TF цепочки `bf → map` в каждом follower-е
- TF цепочки `bf → robot{peer}/map` через `robot{peer}/map`
- Корректной работы SLAM или одометрии без дрейфа

На длинных дистанциях глобальная одометрия дрейфует, и `peer_x, peer_y`
становится менее надёжной, чем относительные lidar-измерения.

**Предложение.** Перейти на body-frame координаты:
```
Telemetry.msg:
  float32 dx          # расстояние до peer (m)
  float32 peer_phi    # азимут peer в bf follower-а (rad)
  float32 v_rel       # продольная относительная скорость (m/s)
  float32 v           # собственная продольная скорость (m/s)
  bool is_valid
```

ACC MPC использует только `dx, v, v_rel` — все они нативно живут в bf.
Глобальные координаты не нужны для swarm-управления (нужны только pacemaker-у
для следования waypoints — у него остаются).

**Преимущества:**
- Robust к дрейфу одометрии
- Не нужен SLAM / `map` frame для follower-ов
- Меньше TF lookup-ов
- Семантически чище

**Что меняется:**
- `Telemetry.msg` — новые поля, удалить `x, y, theta, peer_x, peer_y`
- `peer_localization.py` — упрощается, убирается `bf_to_global`
- `swarm_logic.py` (sliding mode) — переписать на bf-координаты
- Notebooks — поменять колонки для plot-ов траекторий
- RViz — peer как маркер в `bf` follower-а, не в `map`

**Приоритет:** средний.

**Зависит от:** базовый ACC MPC должен сначала заработать в текущей
архитектуре. Рефакторинг после стабилизации.

### 2. Multi-robot DDS networking (реальное железо)

**Контекст.** В симуляторе все ноды живут в одном DDS instance — топики
`/robot1/odom`, `/robot2/odom` видны со всех ROS-нод. На реальном железе
каждый робот — свой compute (Raspberry Pi или аналог), и DDS должен быть
настроен так, чтобы соседние роботы видели топики друг друга.

**Предложение.** Варианты:
- **ROS_DOMAIN_ID**: одинаковый на всех роботах + общий WiFi. Простой, но
  default DDS (Cyclone, Fast-RTPS) плохо работает по WiFi с потерями.
- **Zenoh / `rmw_zenoh`**: более надёжный transport для multi-robot, лучше
  переносит latency и packet loss.
- **DDS bridge**: для соединения сегментов сети.
- **Mesh networking** (например LoRa для critical signals): для безопасности
  при потере основной сети.

**Приоритет:** высокий — необходимо для фазы 7 (тест на железе).

**Зависит от:** наличие физических роботов с настроенным compute.

## Алгоритмические

### 3. Kalman filter для peer state

**Контекст.** Сейчас `peer_v` берётся из odom лидера (точно), а
`peer_x, peer_y` — из lidar-наблюдений (зашумлено). Это два независимых
источника данных без перекрёстной валидации.

**Предложение.** Kalman filter с состоянием `(peer_x, peer_y, peer_vx,
peer_vy)`:
- **prediction step**: CV-модель `peer_pos += peer_v · dt`
- **update step**: lidar (всегда) + odom лидера (если доступно) + own_v_self
  (для compensating self motion в bf-варианте)

Даёт оптимальное по шуму объединение источников, гладкие оценки и явное
управление uncertainty.

**Приоритет:** средний.

**Зависит от:** базовый MPC работает; перед использованием в production-сценариях.

---

### 4. CA-модель лидера

**Контекст.** В предсказании MPC принято `v̇_peer = 0` (constant velocity).
При резких ускорениях/торможениях pacemaker предиктор отстаёт на один цикл
MPC (≈ 50 мс).

**Предложение.** Расширить state до `[dx, v, v_rel, F, a_peer]`, где
`a_peer` — measured disturbance из numerical derivative `peer_v` или из
второй производной `peer_pos`. Динамика `a_peer = const` на горизонте; вход
от лидара/одометрии обновляет state.

**Когда нужно:** только если CV-предположение даст заметные ошибки на
типовых сценариях (резкие старт-стопы pacemaker).

**Приоритет:** низкий — пробовать только если базовый MPC не справляется.

**Зависит от:** базовый MPC, метрики на симуляторе (видно когда он плохо
отрабатывает разгоны pacemaker-а).

---

### 5. Hard safety constraint `dx ≥ d_safe`

**Контекст.** Сейчас безопасный гэп обеспечивается косвенно — через большой
вес `q_gap` в cost-функции. Нет гарантии, что `dx ≥ d_safe` физически.

**Предложение.** Hard inequality constraint в QP:
```
dx_k ≥ d_safe  ∀k ∈ [1, p]
```
Или soft с slack-переменной (большой штраф):
```
min ... + ρ · sum(slack_k²)
dx_k + slack_k ≥ d_safe
slack_k ≥ 0
```

Soft вариант даёт MPC возможность временно нарушить constraint (например,
при exception-сценарии), но штрафуется значительно.

**Приоритет:** средний — для production-сценариев и защиты на железе.

**Зависит от:** базовый MPC; QP-инфраструктура поддерживает linear constraints
(она уже это умеет в OSQP).

---

### 6. Lateral MPC / 2D MPC

**Контекст.** Сейчас угловая скорость задаётся отдельным геометрическим
контроллером по азимуту peer (`u_w = -kp_θ · sign(σ_side)` в swarm_logic).
Это работает, но не использует знаний о траектории / динамике.

**Предложение.** Расширить MPC до 2D с состоянием
`[..., θ_err, ω, dy_lateral]` и моделью bicycle / unicycle. Контроллер
управляет одновременно `v_cmd` и `ω_cmd`.

**Приоритет:** низкий — большой отдельный проект, не критичен для основной
ACC задачи.

**Зависит от:** ACC MPC должен быть стабильным; нужны эксперименты на
поворотах.

---

### 7. CC MPC для pacemaker

**Контекст.** Pacemaker сейчас едет с константной скоростью или по
pure-pursuit. Нет MPC на его продольное управление.

**Предложение.** CC (Cruise Control) MPC с состоянием `[v, F]` без `dx`,
`v_rel`. Вход — целевая скорость `v_ref(s)` по пути. Преимущество — плавный
профиль скорости в поворотах, плавный старт/стоп.

**Класс:** `SwarmCcController` рядом с `SwarmAccController`, общая
инфраструктура (Newton + force-lag), разный output-вектор.

**Приоритет:** низкий — pacemaker не критичен для swarm-стабильности.

**Зависит от:** ACC MPC отлажен; общая инфраструктура классов.

---

### 8. Auto-tuning весов `Q, S, φ`

**Контекст.** Веса cost-функции подбираются вручную, итеративно сравнивая
метрики (rms_jerk, rms_err, η). Это затратно по времени.

**Предложение.** Optimization цикл: запустить N сценариев в симуляторе с
разными Q/S/φ из заданной сетки или с использованием Bayesian optimization
(`scikit-optimize`). Целевая функция — взвешенная сумма метрик.

**Приоритет:** низкий — ручной тюнинг работает на этапе разработки.

**Зависит от:** базовый MPC, repeatable симуляционные сценарии.

## Реализационные

### 9. `drive_controller`: включить публикацию `twist` в odom

**Контекст.** В `drive_controller.cpp` блок вычисления `twist.twist.linear.x`
закрыт под `#if false` (строки 275–285). Сейчас `/robot{N}/odom.twist`
публикуется пустым.

Для MPC на железе это критично — `v_actual` нужен как input. Сейчас
получается только из численного дифференцирования `cmd_vel_back_l/r` или
позиции odom.

**Предложение.**
1. Дописать функцию `calcInverseKinematics` (упомянутую в комментарии): из
   ω левого и правого колеса посчитать `(v, w)` для робота:
   ```
   v = (ω_l + ω_r) · r / 2
   w = (ω_r − ω_l) · r / l
   ```
2. Получить ω колёс из топиков `cmd_vel_back_l/r` от STM32 (это измеренная
   угловая скорость, уже есть)
3. Включить блок: `odom.twist.twist.linear.x = v`, `odom.twist.twist.angular.z = w`

**Приоритет:** высокий для фазы 7 (тест на железе).

**Зависит от:** ничего, можно делать параллельно с фазой 2.

### 10. Симулятор: переход на 2nd-order динамику

**Контекст.** Сейчас симулятор моделирует velocity lag первым порядком:
```
v += α · (v_cmd − v),  α = 1 − exp(−dt/τ_motor)
```
Реальный плант с PID-петлёй STM32 имеет 2nd-order отклик с overshoot/звоном,
хоть и слабым (передемпфированный).

**Предложение.** Перейти на 2nd-order с параметрами `ω_n, ζ`:
```
a += dt · (ω_n² · (v_cmd − v) − 2ζω_n · a)
v += dt · a
```
Параметры в `params_simulator.yaml`. Идентифицировать на реальном роботе
(тот же step-response эксперимент, что для MPC) и подставить в симулятор —
получим симулятор, близкий к железу.

**Приоритет:** средний — после фазы 6 (идентификация на реальном роботе).

**Зависит от:** идентифицированные параметры с железа.

### 11. Симулятор: прямая публикация `a_actual`

**Контекст.** Симулятор знает `v` точно на каждом шаге, но в Odom-сообщении
ускорение не публикуется — приходится численно дифференцировать.
Численное дифференцирование шумит, особенно при высокой частоте
cmd_vel-публикаций (см. фикс в `add_derived` notebook-ов).

**Предложение.** Симулятор вычисляет `a = (v_new − v_old) / dt` и публикует
либо в дополнительном поле, либо отдельным топиком (например
`/robot{N}/acceleration`, `geometry_msgs/AccelStamped`).

Альтернатива: сохранить интерфейс odom, но добавить поле в кастомное
сообщение `RobotState`.

**Приоритет:** низкий — после фикса в notebooks ускорение и так получается
адекватным; ценно только для real-time потребителей.

**Зависит от:** ничего.

### 12. Recovery-логика при потере связи

**Контекст.** Сейчас при `is_valid=False` (peer не виден или TF не
работает) follower публикует нулевой Twist (мгновенная остановка). Это
безопасно, но грубо: при кратковременных потерях лидара робот резко
тормозит.

**Предложение.** Расширенная state-машина:
- `RUNNING` — нормальная работа, MPC решает QP
- `DEGRADED` — peer временно потерян, продолжаем плавно тормозить с
  заданным `a_brake` (использовать last известный `v_rel`)
- `STOPPED` — peer не виден дольше timeout, полная остановка
- Восстановление: переход обратно в `RUNNING` после `dwell_time` стабильных
  валидных измерений

**Приоритет:** средний — для production / реального робота.

**Зависит от:** базовый MPC; параметры timeout-ов нужно подобрать.

---

### 13. Soft start / плавный переход

**Контекст.** При переключении `start: false → true` follower сразу
получает полный output MPC. Это может быть резко.

**Предложение.** Ramp-up: ограничить `u` от 0 до `u_mpc` за `t_ramp` секунд
после включения. Реализуется в ноде, не в контроллере.

**Приоритет:** низкий.

**Зависит от:** базовый MPC.

---

### 14. Fallback `peer_v` через дифференцирование лидара

**Контекст.** В фазе 1 `peer_v` берётся из odom лидера. Если odom долго не
приходит (timeout > 0.5с), сейчас просто `is_valid=False` → остановка.

**Предложение.** При потере odom лидера переключаться на численную
производную `peer_x, peer_y` (как изначально планировалось) с большим LPF.
Это не оптимально, но позволяет продолжать работу при кратковременных
сетевых проблемах.

**Приоритет:** низкий — резервный механизм для увеличения надёжности.

**Зависит от:** базовый peer_localization.

---

### 15. Угловая проекция `peer_v` через `cos(θ_peer − θ_follower)`

**Контекст.** В фазе 1 `peer_v = odom.twist.linear.x` лидера, без
проекции на forward-axis follower-а. На прямом участке разница
пренебрежима, на поворотах — до нескольких процентов.

**Предложение.** Проекция:
```
peer_v_long = peer_v · cos(θ_peer − θ_follower)
```
Требует доступа к `θ_peer` (из odom лидера) и `θ_follower` (из своей odom).

**Приоритет:** низкий — для платуна на гладких траекториях не нужно.

**Зависит от:** базовый peer_localization.
